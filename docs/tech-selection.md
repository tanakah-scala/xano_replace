# 技術選定ドキュメント

## 概要

サービスを Xano から置き換えるにあたり、バックエンド言語・フレームワークの選定を行いました。本ドキュメントはその比較検討の記録です。

> **このリポジトリは個人の学習・検証用です。業務リポジトリとは独立しています。**

---

## バックエンド言語・フレームワーク比較

| 観点 | 採用: Go + Echo | 比較対象: Node.js + Express/Fastify |
|------|----------------|--------------------------------------|
| 学習コスト | 中（静的型・goroutine 等の概念） | 低（フロントと言語統一） |
| パフォーマンス | 高（コンパイル後の高速実行） | 中（V8 ベース、依然高速） |
| 型安全性 | 高（言語標準） | 中〜高（TypeScript で補完） |
| デプロイ容易性 | 高（シングルバイナリ） | 中（node_modules を含む） |
| エコシステム | 中（フレームワークは少なめ） | 高（npm エコシステム） |
| 並行処理 | 高（goroutine） | 中（イベントループ） |
| メモリ使用量 | 低 | 中 |
| 採用しやすさ | 中（Go エンジニア比率は低め） | 高 |

## 採用根拠：Go + Echo v4

### 1. シングルバイナリでの運用容易性

Go はコンパイル時に単一のバイナリを生成します。`node_modules` のような依存関係を持たないため、Docker イメージが軽量になります。マルチステージビルドを利用することで、最終イメージを Alpine ベースの数十 MB に抑えることができます。

```
# Node.js の場合（概算）
イメージサイズ: ~500MB（node_modules 込み）

# Go の場合（概算）
イメージサイズ: ~20MB（Alpine + バイナリのみ）
```

### 2. 型安全性

Go は静的型付け言語であり、コンパイル時に型エラーを検出できます。TypeScript と異なり、型安全性は言語仕様として保証されており、設定不要で恩恵を受けられます。

### 3. 並行処理性能（goroutine）

goroutine は OS スレッドより軽量で、数千〜数万の並行処理を低メモリで実現できます。対象サービスが扱う非同期処理（通知送信・外部 API 呼び出し等）に適しています。

### 4. ECS Fargate / Cloud Run との相性

- シングルバイナリによるコンテナイメージの軽量化
- 起動時間が短い（cold start が短い）
- `GET /health` ヘルスチェックエンドポイントで ECS ターゲットグループのヘルスチェックに対応

## フレームワーク選定：Echo v4

Go の HTTP フレームワークは複数ありますが、Echo を選定した理由：

| フレームワーク | 特徴 |
|---------------|------|
| **Echo v4** | 軽量・高速・ミドルウェア豊富・ドキュメント整備 |
| Gin | 高速・シェア高い。Echoと同等 |
| Chi | 標準 `net/http` 準拠・シンプル |
| Fiber | Express ライク・高速だが独自の実装が多い |

Echo は Gin と並ぶ定番フレームワークで、ミドルウェア（CORS・Logger・Recover）が標準で揃っており、ルーティングがシンプルに記述できます。

## データベース選定：PostgreSQL 16

| 観点 | PostgreSQL | MySQL | SQLite |
|------|-----------|-------|--------|
| 本番実績 | 高 | 高 | 低（組み込み向き） |
| JSON サポート | 高（jsonb） | 中 | 低 |
| 型の豊富さ | 高 | 中 | 低 |
| AWS/GCP 対応 | Aurora PostgreSQL / Cloud SQL | Aurora MySQL / Cloud SQL | — |

Xano のバックエンドは PostgreSQL ベースであるため、移行のしやすさを考慮して PostgreSQL を採用しました。

---

## フロントエンド選定：Next.js 15

既存の cook-share が Next.js ベースであるため、技術スタックの親和性・学習コストの観点から Next.js 15 (App Router) を採用しました。

- `output: 'standalone'` 設定でコンテナイメージを軽量化
- Tailwind CSS でスタイリングを統一
- `NEXT_PUBLIC_API_BASE_URL` 環境変数でバックエンド接続先を切り替え可能

## 今後の検討事項

- **ORM の導入**: `sqlx` や `ent` (entgo.io) によるクエリの型安全化
- **マイグレーションツール**: `golang-migrate` による DB スキーマ管理
- **構造化ログ**: `zerolog` または `slog`（Go 1.21 標準）の導入
- **認証**: JWT（`golang-jwt/jwt`）または Cookie セッション
- **設定管理**: `viper` または `envconfig` による環境変数管理
